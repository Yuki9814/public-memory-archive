# 公共事件长记忆档案馆产品优化计划

版本：2026-05-31  
视角：产品经理挑刺版  
目标：把当前“可展示的档案原型”推进为“可信、可运营、可复核、可规模化扩展的公共事件证据档案产品”。

## 1. 产品定位复盘

### 1.1 当前产品一句话

面向互联网公共争议事件，长期保存公开资料、传播时间线、主张差异、证据等级、修订历史、纠错与举报入口的中立档案系统。

### 1.2 当前最大矛盾

产品底层目标是“严肃、克制、可追溯”，但前台部分语言和信息组织仍带有“围观型产品”的影子：

- README 定位强调“不做热点社区、评论区或性别阵营站”，并要求区分事实、说法、证据、争议和阶段性结论。
- 前端首页却使用“圈内风波簿”“今日小风波候选”“讨论度量榜”等表述。虽然已有免责声明，但“风波”“候选”“榜”天然接近热点消费语境。
- 后端已经提供发布预检、修订快照、纠错、举报、抓取、AI 建议等严肃能力；前端关键提交流和后台工作台仍处于模拟态。

结论：产品真正稀缺价值不在“收集争议”，而在“把争议拆成可复核证据结构”。后续所有体验优化都应围绕“证据结构化”和“风险降噪”展开。

## 2. 核心使用人群与任务

### 2.1 公共访客

任务：

- 快速理解某个事件目前处于哪一阶段。
- 明确哪些内容有强证据，哪些只是说法或弱线索。
- 找到原始资料、存档资料、修订记录。
- 不被页面引导去站队、骚扰或二次传播隐私。

痛点：

- 首页“候选”和“榜”会让访客误以为这是热点索引。
- 搜索和筛选目前主要在前端 20 条列表上做，增长后会失真。
- 详情页信息多，但“证据完整度”“争议未决点”“最近变更”还不够突出。

### 2.2 当事相关方与风险举报者

任务：

- 提交纠错、隐私举报、未成年人保护线索。
- 获得明确回执和处理状态。
- 明白哪些内容会被紧急处理，哪些进入普通复核。

痛点：

- 当前提交、纠错、举报表单只是 `setTimeout` 模拟，不写数据库。
- 回执号是前端随机生成，缺少可追踪状态。
- 入口没有承诺处理优先级、SLA、材料格式和补充机制。

### 2.3 编辑员与馆长

任务：

- 新建事件、补来源、跑存档、抽时间线、抽主张、关联证据。
- 根据发布预检逐项修复。
- 处理线索、纠错、举报，并留存审计记录。

痛点：

- 后台页面多数是模拟面板，没有接入真实 `/admin/*`。
- 后端缺少查看 submissions/corrections 的后台列表与处理接口。
- 发布预检返回机器错误，但前端没有可操作的修复路径。
- AI 接口只返回建议，缺少“引用来源、差异预览、采纳后审计”的编辑体验。

### 2.4 研究者与长期订阅读者

任务：

- 订阅事件修订。
- 比较不同版本变化。
- 跨事件检索主题、平台、证据类型。

痛点：

- RSS 已有，但页面没有明显订阅入口。
- 修订记录只有摘要，没有版本 diff。
- 缺少真正的跨来源、跨主张、跨证据搜索。

## 3. 北极星指标与护栏指标

### 3.1 北极星指标

可复核档案完成率：

```
已发布且满足以下条件的事件数 / 已发布事件总数
```

完成条件：

- 至少 3 个来源或明确说明来源不足。
- 至少 1 条时间线绑定来源。
- 每个关键主张有证据链接，或标为证据不足。
- 有版本快照。
- 纠错入口、举报入口保持开启。
- 无高风险隐私命中。

### 3.2 护栏指标

- 隐私高危举报首响时间：P0 类型 2 小时内标记为 `TRIAGED`。
- 发布预检失败率：高于 30% 时说明编辑端引导不足。
- 纠错采纳后版本生成率：应接近 100%。
- 公开页面动员词命中数：应为 0。
- D 级弱线索单独支撑结论数：应为 0。
- 存档成功率：来源抓取任务成功数 / 来源抓取任务总数。
- API 失败静默降级率：生产环境应接近 0，不能无提示落到 mock。

## 4. 关键问题挑刺

### P0-1：首页语言削弱严肃定位

现象：

- 公开定位是“公共事件长记忆档案馆”。
- 前端品牌与标题出现“圈内风波簿”“今日小风波候选”“讨论度量榜”。

风险：

- “小风波”会轻化线下骚扰、隐私泄露、未成年人相关事件。
- “榜”会把公开来源数量误读成热度、重要性或站队排名。
- 品牌会从“证据档案”滑向“争议消费”。

优化：

- 主品牌统一为“公共事件长记忆档案馆”，副标题可保留“互联网公开资料证据档案”。
- “今日小风波候选”改为“待核验线索池”。
- “讨论度量榜”改为“资料完整度与更新度”。
- 所有事件卡优先展示“资料状态”，弱化序号排名。

验收：

- 首页无“榜”“小风波”“围观”等容易娱乐化的核心标题。
- 卡片展示“来源数、平台数、时间线节点、最近修订”，不使用排名号召。

### P0-2：提交、纠错、举报表单未连接真实 API

现象：

- `apps/web/src/App.tsx` 的 `FormStub` 使用前端状态模拟提交。
- `apps/api/src/routes/public.ts` 已提供 `POST /api/submissions`、`POST /api/corrections`、`POST /api/reports`。

风险：

- 访客以为已经提交材料，实际数据没有入库。
- 隐私和未成年人举报无法触发后台优先级。
- 产品信任受损。

优化：

- 在 `apps/web/src/api.ts` 增加 `createSubmission`、`createCorrection`、`createReport`。
- 表单按后端 schema 提交真实 payload。
- 成功页展示后端返回的 `id` 与 `status`。
- 失败页展示可理解的错误文案，并保留未提交内容。
- 事件详情页纠错/举报 CTA 携带 `eventId`，平台外链举报携带 `platformLinkId`。

验收：

- 提交线索后数据库出现 `Submission`。
- 纠错后数据库出现 `Correction`。
- 举报未成年人或人肉类型后 `priority=10`。
- 网络失败不会生成假回执。

### P0-3：后台工作台停留在演示态

现象：

- 后台页面写着“虚拟映射”“模拟审核”。
- `reviewTasks` 是前端本地数组。
- 后端已经有事件、来源、平台外链、时间线、主张、证据、发布、下架、举报处理接口。

风险：

- 编辑员无法完成真实建档闭环。
- 发布预检无法被转化为具体修复任务。
- 审核、发布、下架缺少可视化风险确认。

优化：

- 后台拆成 5 个真实工作区：事件草稿、来源与存档、证据与主张、发布预检、举报与纠错。
- 事件编辑页接入 `POST /admin/events` 与 `PATCH /admin/events/:id`。
- 来源页接入 `POST /admin/events/:id/sources`、`POST /admin/events/:id/platform-links`、`POST /admin/sources/:id/capture`、`GET /admin/tasks/:taskId`。
- 发布页接入 `POST /admin/events/:id/review` 与 `POST /admin/events/:id/publish`，展示 `failedChecks` 对应字段和修复入口。
- 举报页接入 `GET /admin/reports` 与 `POST /admin/reports/:id/resolve`。

验收：

- 从后台可创建草稿、添加来源、触发抓取、提交审核、发布或查看预检失败。
- 每个发布失败项可跳转到对应编辑区。
- 举报处理结果写入数据库，并产生 audit log。

### P0-4：线索与纠错没有后台处理闭环

现象：

- 公共 API 能创建 submissions 与 corrections。
- 后台仅有 review tasks 和 reports 列表，缺少 submissions/corrections 的查看、分派、采纳、驳回、转事件草稿接口。

风险：

- 访客输入材料后进入数据黑洞。
- 编辑团队无法高效消化外部线索。
- 纠错采纳后无法自动生成版本快照。

优化：

- 新增 `GET /admin/submissions`、`POST /admin/submissions/:id/resolve`。
- 新增 `GET /admin/corrections`、`POST /admin/corrections/:id/resolve`。
- 纠错采纳时要求关联 event/source/timeline/claim/evidence/platformLink 中至少一个对象。
- 采纳纠错后自动生成 `EventVersion`，状态置为 `ACCEPTED` 或 `RESOLVED`。
- 驳回时保留内部原因，前台只展示克制状态。

验收：

- 外部提交能进入后台队列。
- 纠错从 `PENDING` 到 `ACCEPTED/REJECTED/RESOLVED` 全链路可追踪。
- 采纳纠错必然产生版本快照。

### P0-5：前端数据降级策略会掩盖生产故障

现象：

- `apps/web/src/api.ts` 多个 fetch catch 后直接返回 mock 数据。

风险：

- API 挂掉时访客看到旧假数据，误以为档案仍正常。
- 管理员无法发现接口故障。
- 搜索、详情、来源等页面可能展示演示事件。

优化：

- 区分开发环境与生产环境。
- 开发环境可 mock，生产环境应展示错误态和重试按钮。
- 首页和详情页显示“当前为离线演示数据”标识。
- 增加 API 健康状态提示。

验收：

- 生产构建中 API 失败不会静默展示 mock。
- 开发环境 mock 有清晰水印。

## 5. P1 产品能力优化

### P1-1：事件列表应接入服务端查询

现状：

- `GET /api/events` 已支持 query、topic、tag、eventProcessStatus、platform、dateFrom、dateTo、sort。
- 前端目前只拉 `pageSize=20`，再做本地搜索、议题、状态筛选。

优化：

- 事件列表页筛选项与 URL query 双向同步。
- 每次筛选调用服务端 `GET /api/events`。
- 增加平台、标签、日期区间、排序、分页。
- 搜索页升级为单独 API 或复用事件列表 query。

验收：

- 第 21 条之后的事件可被筛选与搜索命中。
- 刷新页面后筛选条件保留。
- 空结果说明区分“无资料”和“筛选过窄”。

### P1-2：详情页要更像“证据报告”，而非信息堆叠

优化方向：

- 顶部增加“档案可信度摘要”：来源数、强证据数、弱线索数、未决关键主张数、最近修订。
- 时间线每个节点展示来源等级、来源标题、是否存档、是否仍可访问。
- 主张矩阵按“已证实、部分支持、证据不足、有争议、已驳回”分组。
- 证据柜增加筛选：证据等级、证据类型、来源平台、是否存档。
- 修订历史增加“本次新增/修改/删除”摘要。
- 纠错和举报 CTA 应贴近具体模块，例如来源卡、主张卡、时间线节点旁边可直接进入对应对象的反馈表。

验收：

- 新访客 30 秒内能判断“当前可确定内容”和“仍有争议内容”。
- 每条关键叙述都有可点击来源锚点。
- 举报入口能携带事件、来源或外链上下文。

### P1-3：发布预检要产品化

现状：

- 后端 `runPublishPreflight` 检查中性标题、来源、时间线来源绑定、证据、隐私、敏感信息、版本、纠错与举报入口。
- 返回错误偏工程化，前端没有修复指导。

优化：

- 后台发布页显示“发布前体检报告”。
- 每个 failed check 对应中文解释、风险级别、修复入口。
- 支持“一键定位字段”。
- 支持“再次运行预检”。
- 预检通过后仍展示二次确认，提醒发布责任。

验收：

- 发布失败不再只显示 JSON。
- 编辑员能从失败项直接跳到相关字段。
- 预检通过记录在 audit log 或事件版本摘要中。

### P1-4：抓取与存档状态需要可见

现状：

- worker 可抓取 HTML、hash、Wayback。
- Prisma 中有 screenshotUrl，但 worker 暂未生成截图。
- 前端仅展示 `archiveUrl` 与 `capturedAt`，缺少任务过程。

优化：

- 来源管理页显示抓取任务状态：queued、active、completed、failed、progress。
- 抓取完成后展示 hash、finalUrl、waybackUrl、下一次复抓时间。
- 失败时显示 HTTP 状态、重试按钮、人工备注。
- 如需封面图，补截图生成能力或明确仅用平台缩略图。

验收：

- 编辑员能判断某条来源是否已完成存档。
- 失败抓取能重试并保留错误。
- 公开页不把未存档资料包装为已核验。

### P1-5：举报处理要有优先级和公开回执语义

优化：

- 举报类型分 P0/P1/P2：未成年人、人肉、隐私泄露最高。
- 回执页展示“已进入高优先级复核”而非“紧急审查程序启动”这类略夸张措辞。
- 后台队列按 priority、status、createdAt 排序。
- 处理时可选择：隐藏字段、隐藏外链描述、下架事件、驳回、转纠错。
- 处理完成后可生成公开状态，不暴露举报人信息。

验收：

- `MINOR_INFO`、`DOXXING` 自动置顶。
- 每次处理有 `resolutionNotes`。
- P0 举报可先隐藏风险字段，再继续完整复核。

## 6. P2 增长与长期能力

### P2-1：待核验线索池产品化

当前候选池是前端静态数组，适合 demo，不适合运营。

优化：

- 新增 Candidate 模型：标题、社区、来源数、平台数、状态、风险等级、是否公开展示。
- 候选公开展示前必须经过隐私预检。
- 候选卡只展示抽象议题，不展示可识别当事信息。
- 支持从 submission 一键转 candidate 或 event draft。

验收：

- 候选池不再依赖前端 mock。
- 高风险候选默认不公开。

### P2-2：AI 辅助从“假智能”变成“审稿副驾驶”

现状：

- AI 路由实际上基于规则和已有字段生成建议。
- 返回 `suggestions_only` 是正确边界，但缺少可信度、引用与 diff。

优化：

- AI 建议必须带来源引用、字段目标、风险提示、置信度。
- 编辑端提供“预览差异”“采纳为草稿”“忽略并记录原因”。
- AI 中立性检查覆盖标题、摘要、时间线、主张、平台外链描述。
- 所有 AI 采纳动作写入 audit log，且不能直接发布。

验收：

- AI 建议无来源不得采纳。
- 采纳后仍需发布预检通过。

### P2-3：版本 diff 与订阅能力增强

优化：

- 修订记录展示字段级 diff。
- 每个事件详情页提供 RSS 链接。
- 支持“最近变化”模块：新增来源、修改摘要、处理举报、状态变化。
- 可导出事件 JSON，供研究者复核。

验收：

- 访客能看懂每个版本变更。
- 研究者能订阅事件更新。

### P2-4：平台治理与法律边界

优化：

- 增加公开页面：收录标准、移除标准、纠错规则、隐私政策、版权说明、联系方式。
- 明确不收录内容：未成年人可识别信息、普通人住址联系方式、鼓励骚扰材料、无公共利益的私人纠纷。
- 增加“平台原帖不可访问时的处理规则”。

验收：

- 法律与伦理边界在页面可见。
- 举报处理可引用公开规则。

## 7. 里程碑计划

### 第 0 阶段：定位修正与真提交闭环，1 周

目标：

- 修正文案风险。
- 表单接入真实公共 API。
- 生产环境禁止静默 mock。

任务：

- 替换首页“风波/候选/榜”相关标题。
- 新增公共表单 API helper。
- 改造 `FormStub` 为真实提交组件。
- 表单成功页使用后端返回 id/status。
- API catch 按环境区分 mock 与错误态。

验收：

- 三类表单均可写入数据库。
- API 失败不生成假回执。
- 首页无娱乐化核心标题。

### 第 1 阶段：编辑工作台最小可用，2-3 周

目标：

- 后台可以真实建档、补来源、跑预检、发布。

任务：

- 事件草稿列表与编辑表单。
- 来源与平台外链表单。
- 抓取任务状态查看。
- 主张与证据关联。
- 发布预检 UI。
- 举报队列真实接入。

验收：

- 管理员从 0 到发布可完整走通。
- 预检失败项能定位修复。
- 举报处理能落库并审计。

### 第 2 阶段：线索/纠错运营闭环，3-5 周

目标：

- 外部输入材料可被处理、追踪、转化。

任务：

- 新增 submissions/corrections 管理 API。
- 后台线索队列。
- 纠错采纳生成版本。
- submission 转 event draft 或 source。
- 高风险举报字段隐藏能力。

验收：

- 所有外部输入都有后台状态。
- 纠错采纳必然生成版本记录。
- P0 举报可快速处置。

### 第 3 阶段：搜索、订阅、版本 diff，5-8 周

目标：

- 产品从“能看”升级为“能查、能订阅、能审计”。

- 研究者可导出结构化档案。

### 第 4 阶段：候选池与 AI 审稿副驾驶，8-12 周

目标：

- 提升编辑效率，但保持人工责任边界。

任务：

- Candidate 模型与后台管理。
- submission 转 candidate。
- AI 建议加引用、置信度、diff。
- 中立性检查覆盖更多字段。
- AI 采纳审计。

验收：

- 候选池不依赖静态 mock。
- AI 不能绕过人工审核与发布预检。

## 9. 信息架构建议

### 9.1 公共端导航

- 首页
- 档案索引
- 资料检索
- 议题地图
- 方法论
- 纠错与举报

后台入口仅管理员可见，不在访客导航中强化。

### 9.2 事件详情结构

推荐结构：

1. 档案标题与当前阶段
2. 可信度摘要
3. 中立摘要
4. 最近变化
5. 原始资料跳转
6. 时间线
7. 主张矩阵
8. 证据柜
9. 资料源列表
10. 已知事实
11. 争议未决
12. 不应推断
13. 修订历史
14. 纠错与举报

当前组件顺序基本合理，但需要把“可信度摘要”和“最近变化”前置。

### 9.3 后台工作台结构

- 概览：待处理举报、待处理纠错、预检失败草稿、抓取失败来源。
- 事件：草稿、编辑、审核、发布、下架。
- 来源：资料源、平台外链、抓取任务。
- 证据：主张、证据、关联关系。
- 线索：submissions、candidates。
- 安全：reports、隐私处理记录。
- 审计：版本、audit log、AI 采纳记录。

## 10. 风险清单

### 10.1 舆论放大风险

触发点：

- 候选池公开。
- 讨论度量榜。
- 平台外链直接跳转。

控制：

- 改名为资料完整度。
- 不展示热度类互动数，或放入后台。
- 候选高风险默认不公开。

### 10.2 隐私与未成年人风险

触发点：

- 外部提交材料包含身份信息。
- 时间线或主张中出现可识别细节。
- 原始平台链接含未打码截图。

控制：

- 提交材料先进入后台，不直接公开。
- 发布预检增加平台外链 description 与 source summary 检测。
- 公开页提供就地举报入口。

### 10.3 档案可信度风险

触发点：

- D 级线索被误读成事实。
- 当事人陈述被写成结论。
- 媒体转述被当作官方结论。

控制：

- 主张矩阵按证据状态展示。
- 每个事实节点显示来源等级。
- 文案规范纳入发布预检。

### 10.4 技术运营风险

触发点：

- API 故障回退 mock。
- 抓取失败没有重试与告警。
- 后台没有处理队列。

控制：

- 生产错误态显性化。
- 任务状态可视化。
- 外部输入有后台闭环。

## 12. 验收测试方案

### 12.1 P0 验收

- 使用提交线索页提交标题、正文、来源链接，确认数据库存在记录。
- 使用纠错页提交修正说明，确认后台可见。
- 使用举报页提交 `MINOR_INFO`，确认 priority 为 10。
- 断开 API 后访问前端，确认显示错误态而非 mock 档案。
- 管理员登录后可看到真实举报队列。

### 12.2 P1 验收

- 创建事件草稿，添加来源，添加平台外链，触发抓取。
- 发布前缺少时间线来源绑定时，预检失败并定位到时间线。
- 修复后重新预检并发布。
- 公开详情页能看到来源等级与存档状态。
- 事件列表服务端分页，第二页数据可访问。

### 12.3 P2 验收

- 外部 submission 转为 candidate。
- AI 生成主张建议，编辑端预览 diff 后采纳为草稿。
- 版本 diff 页面显示字段变化。
- 单事件 RSS 可在详情页发现。

## 14. 产品经理最终挑刺结论

当前项目最有价值的部分在后端规则、数据模型和内容规范，最薄弱的部分在“把能力变成可信产品体验”。若继续堆首页模块，会越来越像事件聚合站；若优先打通提交、审核、预检、修订、举报闭环，产品会真正建立护城河。

建议近期只围绕一句话推进：

> 任何公开展示的判断，都必须能追到来源、证据、修订和责任动作。

这句话也是后续所有功能取舍的准绳。
